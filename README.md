# Theory-of-Mind Benchmark

A testbed where LLM agents must correctly model another agent's *false* belief — not
just the true state of the world — to succeed at a task. Built to the design in
[`tom-benchmark-prd-mvp.md`](tom-benchmark-prd-mvp.md).

Every scenario is procedurally generated as a **matched pair**: a false-belief trial
(F) where the correct answer requires tracking what a scripted NPC still believes,
and a true-belief control (T) with identical world complexity. The gap between T and
F accuracy is the headline metric — see [`site/methodology.html`](site/methodology.html)
for the full write-up.

**Cost: $0.** No database, no paid hosting, no required API spend. The repo *is* the
database (`data/*.json`, committed), and the demo site is static HTML/JS read by
GitHub Pages. Model calls only ever hit free tiers or a one-time free signup credit,
and the pipeline can be fully exercised with zero API keys via `--mock`.

## Repo layout

```
src/tom_bench/        scenario engine, deterministic Seeker, prompts, scoring, clients, runners
scripts/               CLI entry points (generate scenarios / run batch / compute scores)
data/scenarios/        generated scenario pairs (committed JSON)
data/trials/            per-trial transcripts (committed JSON)
data/scores.json        aggregated leaderboard numbers (regenerated by compute_scores.py)
data/trials_index.json  lightweight manifest the replay page uses to browse trials
data/reference_corpus/  small fixture corpus for the novelty-check heuristic
site/                    static demo site (leaderboard, replay viewer, methodology) — GitHub Pages
tests/                   pytest suite
```

## Setup

Requires Python 3.11+.

```bash
python -m venv .venv
.venv/Scripts/activate        # Windows; use `source .venv/bin/activate` on macOS/Linux
pip install -r requirements.txt
pytest -q                     # 42 tests, no network/API keys required
```

## Running the pipeline

**1. Generate scenarios** (procedural, reproducible from a seed, novelty-checked):

```bash
python scripts/generate_scenarios.py --n 50
```

**2. Run the batch, at zero cost, with no API keys** — validates the entire pipeline
end-to-end using a deterministic mock client that never calls a real API:

```bash
python scripts/run_batch.py --mock
```

**3. Compute the leaderboard:**

```bash
python scripts/compute_scores.py
```

That's the full loop with **zero API keys and zero cost** — useful for development,
CI, and confirming the pipeline works before spending any free-tier quota.

### Running against real free-tier models

1. Copy `.env.example` to `.env` and fill in whichever free-tier keys you have. You
   don't need all three — any model with no key set is skipped automatically.
   - `GEMINI_API_KEY` — https://aistudio.google.com/apikey (permanent free tier, no card)
   - `GROQ_API_KEY` — https://console.groq.com/keys (permanent free tier, no card)
   - `ANTHROPIC_API_KEY` — https://console.anthropic.com (one-time free signup credit, no card charge)

   `.env` is gitignored — never commit real keys.

2. **Always dry-run first** to see how many API calls a batch will actually make,
   before it makes any:

   ```bash
   python scripts/run_batch.py --dry-run
   ```

3. Run a small pilot batch against real APIs before committing to the full run:

   ```bash
   python scripts/run_batch.py --limit-pairs 5
   ```

4. Run the full batch. It writes trials incrementally and skips any trial that
   already has a file on disk, so if a free-tier daily quota gets hit partway
   through, just re-run the same command later (same day or the next) to resume:

   ```bash
   python scripts/run_batch.py
   python scripts/compute_scores.py
   ```

Free-tier rate limits and model names shift over time — the client modules in
`src/tom_bench/clients/` document current defaults and the env vars to override them
(`GEMINI_MODEL`, `GROQ_MODEL`, `ANTHROPIC_MODEL`, and matching `*_MIN_INTERVAL_SECONDS`
pacing overrides). Confirm current limits at each provider's dashboard before a real
run, per the PRD's explicit caveat.

## Viewing the demo site locally

The site fetches committed JSON via relative paths, so it needs to be served over
HTTP (not opened as a `file://` URL):

```bash
python -m http.server 8000
# then open http://localhost:8000/site/index.html
```

## Publishing to GitHub Pages (optional, still $0)

1. Push this repo to GitHub.
2. Repo Settings → Pages → Source: **Deploy from a branch** → `main` → `/ (root)`.
   (Root, not `/docs` — the site fetches `../data/...`, so both `site/` and `data/`
   need to be published together.)
3. The leaderboard will be live at `https://<you>.github.io/<repo>/site/index.html`.

## Tests

```bash
pytest -q
```

Covers: scenario reproducibility from a seed, F/T pair complexity-matching, the
novelty check (including that it correctly flags an inserted classic vignette),
Seeker logic, answer parsing/scoring/failure-tagging edge cases (including
unparseable output), Wilson CI bounds, score aggregation, model client payload/parse
functions, and batch-runner resumability under a simulated mid-batch API failure.

## Privacy / secrets

No API keys, tokens, or personal data are committed anywhere in this repo. `.env` is
gitignored; only `.env.example` (with blank placeholders) is tracked. Model client
code never logs key values — only which provider was configured or skipped.
