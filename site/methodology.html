Theory-of-Mind Benchmark

Leaderboard Trial replay Methodology

This page exists to answer one question a technically literate reader should ask: how do I know this isn't just Sally-Anne with new names?

The task

Each scenario is a small procedurally-generated household world (3–4 rooms, containers, everyday objects). A target object is placed in room P while a scripted NPC ("the Seeker") watches. The Seeker then leaves for room Q. While the Seeker is away, the object is moved to room R.

The model under test (the Helper) sees the entire event log — including the move the Seeker never witnessed — and must place an assist marker at the room where the Seeker will look first. In the interactive framing this is a forced placement action; in the quiz framing it's a 3-way multiple-choice question generated from the same scenario data, so the two task types never drift out of sync with each other.

Why the control condition matters

Every scenario is generated as a matched pair: a false-belief trial (F) where correct = P (what the Seeker still believes) and a true-belief control (T), built from the identical world and move event, where the Seeker is truthfully told about the move and correct = R (the true, current location). F and T are held to identical room count, object count, and event count — the only thing that differs is whether the Seeker's belief tracks reality.

This is the point of the whole design: a model that just reports the true state of the world will score well on T and badly on F. A model that's actually modeling the Seeker's belief will score well on both. The ToM gap (accuracy(T) − accuracy(F)) is reported per model as the headline number, alongside plain accuracy, because a gap near zero and a gap near one both fail to distinguish "isn't tracking beliefs at all" from "always predicts the last-seen location regardless of condition" — the gap plus both raw accuracies together are what's diagnostic, not the gap alone.

Guarding against contamination

Classic false-belief vignettes ("Sally puts her marble in a basket…") are almost certainly in every frontier model's training data verbatim. A benchmark that reuses that shape risks measuring memorized pattern-completion, not belief modeling. This project avoids the classic vignette's fixed names and objects entirely (no "Sally," "Anne," "basket," or "marble" anywhere in the generator), procedurally randomizes room count, object identity, NPC name, and container choice per scenario from a seed, and renders the narrative through several paraphrase templates per story beat rather than one fixed sentence.

Every generated scenario is additionally run through a lightweight novelty check — trigram (3-word) Jaccard overlap against a small reference corpus of classic false-belief vignette paraphrases — and rejected/regenerated if the overlap exceeds a threshold. This is a heuristic, not a guarantee. It catches gross textual overlap (e.g. a template bug that reproduces the textbook paragraph shape); it does not prove semantic novelty, and a model could in principle still recognize the underlying task structure without lexical overlap. We report this limitation explicitly rather than claiming contamination is solved.

Scoring & failure taxonomy

A model's raw response is parsed for a structured JSON answer; if no valid location can be extracted, the trial is tagged unparseable and excluded from accuracy (not silently scored wrong). Wrong answers are further tagged:

  • used_true_state_instead_of_belief — the model answered with the object's actual current location instead of the Seeker's belief (only meaningful on F trials — the classic "failure to model false belief" signature).
  • picked_neither_location — the model named a room that is neither the believed nor the true location (e.g. the decoy room).
  • wrong_belief_location — a residual bucket for answers that don't fit the above (e.g. wrong on the true-belief control despite naming a plausible room).

Accuracy is reported with a 95% Wilson score confidence interval (more reliable than the normal approximation at this sample size and at accuracy values near 0 or 1).

Free-tier disclosure

This project runs entirely on free-tier infrastructure: Google Gemini (Google AI Studio's permanent free tier), an open-weight model via Groq's free tier, and Anthropic Claude on the one-time free signup credit. Free-tier request/rate limits and model availability change over time; the batch runner records which model identifier answered each trial (see the model_name field on any trial in the replay view), and the leaderboard should be read as "results captured under the free-tier terms in effect at generation time," not a permanent characterization of a model family.

The data currently shown on the leaderboard was generated with the pipeline's mock model client (scripts/run_batch.py --mock) to validate the engine end-to-end at zero cost and with no API keys required. It exercises the full generation → scoring → display pipeline but is not a real model result — treat any numbers you see as a pipeline smoke test until this note is updated to reflect a run against real free-tier model APIs.

Known limitations

  • Tier 1 only (single NPC, single false-belief event, at most one distractor event, first-order belief). Multi-NPC, lied-to, and second-order-belief scenarios are out of scope for this MVP.
  • Small sample size (50 matched pairs at MVP scale) — confidence intervals are reported honestly and should be read as a first data point, not a definitive claim.
  • The novelty check is a heuristic (see above) — not proof of non-contamination.
  • The Seeker's belief-update rule is simple (last personally-observed location, or a fixed room-sweep fallback) — it does not model the Seeker reasoning about anyone else's beliefs in turn.
